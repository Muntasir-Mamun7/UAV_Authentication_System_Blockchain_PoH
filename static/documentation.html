System Architecture

Central blockchain authority managing flight authentication and data integrity

Individual drone instances performing authentication and sending telemetry

Authentication System

5G-AKA Protocol

The system implements a simplified 5G Authentication and Key Agreement (AKA) protocol for mutual authentication between UAVs and the GCS.

Authentication Flow:

  1. UAV requests flight authorization
  2. GCS sends random challenge (RAND)
  3. UAV calculates response (RES*) using long-term key
  4. GCS verifies response and generates session key
  5. Mutual authentication established

User Authentication

Web dashboard users authenticate using JWT (JSON Web Tokens) with bcrypt password hashing.

Security Features:

  • bcrypt password hashing (cost factor: 12)
  • JWT tokens with expiration (24 hours)
  • Session management and token invalidation
  • Secure password requirements

Role-Based Access Control (RBAC)

User Roles

Administrator

Full system access with the following privileges:

  • View all flights from all UAVs
  • Manage user accounts (create, disable, promote)
  • Assign/unassign UAVs to users
  • Access system statistics and logs
  • Retrain AI anomaly detection model
  • Force-end stuck flights

Normal User

Limited access based on UAV assignments:

  • View flights only from assigned UAVs
  • Operate assigned UAVs
  • Export flight logs for assigned UAVs
  • No administrative functions

UAV Assignment Management

Administrators can assign specific UAVs to users, controlling which flights they can access and operate.

Example Scenario:

User: pilot_east
Assigned UAVs: UAV_A1, UAV_B2
Result: Can only view/operate UAV_A1 and UAV_B2 flights

Proof of History Blockchain

The system uses a custom Proof of History (PoH) blockchain implementation to create tamper-proof, chronologically ordered flight records.

Block Structure

{
  "index": 0,
  "timestamp": 1730668800.123,
  "transactions": [...],
  "previous_hash": "abc123...",
  "current_hash": "def456...",
  "event_log": [...]
}

Integrity Verification

The system performs real-time integrity checks on all flight data:

  • Hash Chain Verification: Each block's hash must match the next block's previous_hash
  • Chronological Verification: Timestamps must be strictly increasing
  • Data Integrity: Any tampering with historical data is immediately detected

Tamper Detection

If any block is modified, the hash chain breaks and the system flags the entire flight as "TAMPERED"

Get list of archived flights (RBAC filtered)

All admin endpoints require admin role and valid JWT token

GET /api/admin/users

List all system users

POST /api/admin/assign_uav

Assign UAV to user

PUT /api/admin/user/{username}/role

Change user role

GET /api/admin/system_stats

Get system statistics

Troubleshooting

Cannot connect to GCS

Symptom: "Connection error" in dashboard

Solution:

  • Ensure GCS_LeaderNode.py is running
  • Check that port 5000 is not in use
  • Verify firewall settings

Flight stuck in "Active" state

Symptom: Flight doesn't end or archive

Solution:

  • Login as admin
  • Go to Admin Panel → System Stats
  • Click "Manage Active Flights"
  • Force-end the stuck flight

User cannot see flights

Symptom: Normal user sees "No flights found"

Solution:

  • Check if UAV is assigned to the user
  • Admin must assign UAV via Admin Panel → UAV Assignments
  • User can only see flights from assigned UAVs

AirSim connection failed

Symptom: UAV client cannot connect to simulator

Solution:

  • Ensure Unreal Engine with AirSim is running
  • Check AirSim IP address in UAV client (default: 192.168.43.231)
  • Wait 3 seconds after starting AirSim before running client
